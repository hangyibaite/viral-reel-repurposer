# Question batches

When the user pastes a transcript but hasn't given complete direction, ask the right batch. Never ask one question at a time. Always batch.

Match the scenario below. Use the questions exactly as written, or adapt for the specific transcript.

---

## Scenario 1 — User pasted a transcript with NO direction at all

The user just dropped the transcript. They didn't say what to do with it.

Ask:

1. What concept, tool, or service should replace the original's central topic?
2. Who is the audience — broad top-of-funnel (anyone) or a specific niche (e.g., a particular industry, role, or ICP)?
3. What's the CTA format you want — DM keyword, comment keyword, link in bio, caption pointer, or something else?

---

## Scenario 2 — User gave the topic but no audience or CTA

The user said "rescript this for [topic]" but didn't specify audience or CTA.

Ask:

1. Who is the audience for this — broad top-of-funnel, or a specific niche?
2. What's the CTA format?
3. Is there a target runtime, or should I match the original's length?

---

## Scenario 3 — User gave topic and audience but no CTA

Ask:

1. What's the CTA format?
2. Is the payoff line of the original something you want to preserve, or should I write a new one that ties to your CTA?
3. Any iconic lines from the original you want preserved word-for-word, or any you want killed?

---

## Scenario 4 — User gave everything but the new content is too different from the original

The original is about X and the new topic is so different that direct preservation will break. The user gave clear direction but the gap is wide.

Ask:

1. The original's [SPECIFIC BEAT] doesn't have a natural equivalent in your topic. Do you want to (a) cut that beat entirely, (b) replace it with [SUGGESTED EQUIVALENT], or (c) keep it as a tonal callback even if the content shifts?
2. The original has [SPECIFIC PROOF POINT]. Do you have an equivalent proof point for your topic, or should I swap it for a process claim?

(Limit this to 2 questions. The user has already given enough direction that you should be writing — just resolve the structural mismatches.)

---

## Scenario 5 — User pasted a transcript with timestamps already

Sometimes the user gives you a pre-formatted transcript. Don't re-format unless asked. Skip step 1 of the main workflow.

But still ask the direction questions if direction is unclear (use Scenario 1 or 2).

---

## Scenario 6 — User is mid-iteration on a script you already wrote

The user has already received a draft and is asking for tweaks. Do NOT re-ask the direction questions. Instead:

1. Read their feedback closely.
2. Identify the pattern in what they're changing (preservation harder / different tone / different angle / etc.).
3. Apply the pattern. Output a revised version.
4. If a paraphrase is needed for the change, follow the 3-5 options rule.

If the feedback is genuinely ambiguous, ask ONE clarifying question. Not three.

---

## Scenario 7 — User pasted a long-form transcript (over 90s)

In addition to the standard questions, ask:

1. Is the argument structure intact in the original? (i.e., does the original have a clear claim → setup → evidence → thesis → implication → action flow, or is it more freeform?)
2. If freeform, what's the ONE thing you want the viewer to take away?

---

## Scenario 8 — User wants to repurpose but the original has heavy specifics you can't match

The original contains specific numbers, named clients, dated case studies — and the user is a smaller creator without that proof.

Ask:

1. Do you have equivalent proof for your topic (numbers, results, named clients), or should I swap the specifics for process/principle claims?
2. If process/principle, what's something you've genuinely done that I can reference at a general level?

---

## Hard rules for asking

1. Never ask more than 3 questions in one turn.
2. Never ask one question at a time across multiple turns. Batch.
3. Never ask questions if the user has already supplied the answer somewhere in the conversation.
4. Never ask questions just to delay writing. If you have enough to start, start.
5. After asking, WAIT. Do not write a script until the user has answered.
